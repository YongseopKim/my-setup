# CLAUDE.md 해설서

이 문서는 `~/.claude/CLAUDE.md`의 구조, 설계 의도, 각 섹션 간의 관계를 설명한다.

---

## 문서의 성격

`CLAUDE.md`는 Claude Code가 **모든 프로젝트·모든 세션에서** 로드하는 글로벌 지침서다.
프로젝트별 `CLAUDE.md`(리포 루트)와는 별개이며, 여기에는 프로젝트에 무관하게 항상 적용되어야 할 규칙만 담는다.

---

## 전체 구조

문서는 **라이프사이클 순서**로 배치되어 있다:

```
절대 규칙 → 프로젝트 셋업 → 세션 시작 → 개발 워크플로우 → 개발 중 실천 → 세션 종료 → 소통 원칙
(항상)      (1회)          (매 세션)    (기능 단위)       (상시)         (매 세션)    (항상)
```

이 순서에는 의도가 있다. Claude가 문서를 위에서 아래로 읽을 때, 자연스럽게 "지금 내가 어느 단계에 있는가"를 파악할 수 있도록 배치했다.

---

## 섹션별 해설

### ⛔ Bash Commands

| 항목 | 설명 |
|------|------|
| **왜 최상단인가** | 이 규칙은 다른 모든 행동에 선행한다. 위반 시 사용자가 매번 수동 승인을 해야 하므로, 가장 먼저 인지시키기 위해 최상단에 배치했다. |
| **왜 이렇게 장황한가** | Claude는 `&&`, pipe, 줄바꿈 등을 자주 결합한다. 표, 금지 예시, 올바른 예시를 모두 나열해야 실제로 따른다. 간략한 한 줄 규칙은 효과가 없었다. |
| **Subagent 규칙** | Claude가 Agent tool로 하위 에이전트를 생성하면, 하위 에이전트는 이 CLAUDE.md를 직접 읽지 않는다. 따라서 dispatch 시 규칙을 명시적으로 전달하라는 지시가 필요하다. |

### Project Setup

| 항목 | 설명 |
|------|------|
| **적용 시점** | 사용자가 "새 프로젝트를 만들어줘" 등의 요청을 할 때 1회 적용. |
| **서비스 관리** | 사용자의 개발 환경(macOS)과 배포 환경(Ubuntu)을 모두 고려한다. 리포 안에 서비스 파일을 두는 이유는 서비스 설정이 코드와 함께 버전 관리되도록 하기 위함이다. |
| **.gitignore** | Claude Code와 MCP 서버들이 생성하는 작업 디렉토리가 git에 커밋되지 않도록 방지한다. MCP 서버 목록은 고정값이 아니므로 settings.json을 확인하라는 지시를 포함한다. |
| **MEMORY 초기화** | 프로젝트별 메모리 파일을 만들어 세션 간 연속성을 확보한다. |

### Session Start

| 항목 | 설명 |
|------|------|
| **스코프** | 매 대화 세션의 시작 시점에 실행. |
| **순서가 중요한 이유** | .venv를 먼저 활성화해야 이후 Python 명령어가 올바른 환경에서 실행된다. MEMORY는 그 다음에 읽어야 이전 세션의 맥락 위에서 작업을 시작할 수 있다. |
| **"If it does not exist, create it"** | 기존 프로젝트에 처음 진입하는 경우를 대비한 fallback. 없으면 읽기 실패로 멈추지 않도록 한다. |

### Development Workflow

| 항목 | 설명 |
|------|------|
| **스코프** | 기능 하나 또는 버그 수정 하나 단위. 한 세션에서 여러 번 반복될 수 있다. |
| **TDD 강제** | Step 4에서 RED → GREEN → VERIFY 사이클을 명시한다. 테스트 없이 구현부터 시작하는 것을 방지한다. |
| **Step 6 vs Development Completion** | Step 6은 **기능 단위** 머지/정리이고, Development Completion은 **세션 단위** 마무리다. 의도적으로 분리되어 있다. 한 세션에서 기능 A → 기능 B를 순차 개발할 때, 기능 A의 Step 6 후 바로 기능 B로 진행하고, Completion은 세션 끝에만 실행된다. |

### Development Practices

| 항목 | 설명 |
|------|------|
| **스코프** | 개발 중 항상 적용되는 실천 사항. 특정 단계에 종속되지 않는다. |
| **Worktree 경로 처리** | Claude Code의 permission 시스템은 명령어 패턴 기반이다. `pytest tests/`로 허가받았으면, `/path/to/worktree/pytest tests/`는 다른 패턴으로 인식되어 재승인을 요구한다. `cd` 후 상대 경로를 쓰면 이 문제를 회피할 수 있다. |
| **스킬 활용 기준** | 어떤 스킬을 언제 쓸지의 판단 기준을 제공한다. brainstorming(모호한 요구), writing-plans/executing-plans(큰 기능), code-review(매 계획 단계 완료 후). "매 계획 단계"로 한정한 이유는 TDD 사이클마다 리뷰를 돌리면 과도하기 때문이다. |
| **Bug Fixing** | 독립 섹션이 아니라 Practices 하위에 둔 이유는, 이것이 특정 라이프사이클 단계가 아니라 "개발 중 언제든 적용되는 방법론"이기 때문이다. |

### Development Completion

| 항목 | 설명 |
|------|------|
| **스코프** | 세션/대화 종료 시 1회 실행. 기능 단위가 아니다. |
| **순서의 의도** | ① 스킬 추천 → ② MEMORY 업데이트 → ③ 브랜치 마무리. 스킬 추천을 먼저 하는 이유는, 브랜치를 정리하고 나면 작업 맥락이 사라지기 때문이다. MEMORY를 브랜치 마무리 전에 업데이트하는 이유도 동일하다. |
| **finishing-a-development-branch 스킬** | 이 스킬이 머지, worktree 정리, 브랜치 삭제 등의 세부 절차를 내부적으로 처리한다. CLAUDE.md에서 이를 반복 기술하지 않고 스킬에 위임한다. |

### Communication

| 항목 | 설명 |
|------|------|
| **왜 별도 섹션인가** | 소통 원칙은 라이프사이클에 종속되지 않고 항상 적용된다. 마지막에 두어 "이 모든 과정에서 이렇게 소통하라"는 의미를 부여한다. |
| **핵심** | 사용자가 틀렸다면 맞장구치지 말고 근거와 함께 반박하라. 사용자는 정직한 피드백을 원한다. |

## 설계 원칙

1. **스코프를 명확히 구분한다.** "항상", "프로젝트 1회", "세션마다", "기능마다", "상시" — 각 섹션이 언제 적용되는지 혼동 없이 판단할 수 있어야 한다.
2. **행동 가능한 지시만 담는다.** 철학적 원칙이 아니라 "이 상황에서 이것을 하라"는 구체적 행동을 기술한다.
3. **Claude가 실제로 따르도록 작성한다.** 간결한 한 줄 규칙은 무시되기 쉽다. 위반 사례, 올바른 사례, 판단 기준을 함께 제공해야 준수율이 높아진다 (Bash 규칙이 대표적).
4. **중복을 피한다.** 스킬이 처리하는 세부 절차는 CLAUDE.md에서 반복하지 않고 스킬 이름으로 위임한다.
